

## Executive Summary

### What We Are Building

DealRoom AI is a production-grade, multi-tenant SaaS platform that automates the due diligence process for consulting firms and private equity teams. Before any major acquisition or investment, buyers must investigate the target company by reviewing hundreds of documents covering financials, contracts, and market data. Today this is done manually by teams of junior analysts at a cost of $500K–$2M per engagement, taking four to eight weeks.

DealRoom AI cuts this to minutes. Firms upload documents to a secure, isolated workspace. Three AI agents process them: one indexes every document into a permission-filtered vector store, one autonomously researches the company using live external tools, and one synthesises everything into a structured due diligence report with ML-scored financial risk and full source citations. The report moves through a governed review workflow — junior analyst annotates, senior analyst resolves disputed findings and approves — before anything leaves the platform.

Every action from login to export is written to an append-only audit trail. Documents are watermarked on download. Access is controlled at tenant, deal room membership, and individual document levels — meeting the access control standard expected of enterprise VDR (Virtual Data Room) platforms.

### Target Users

| Persona | Role in Platform | Primary Need |
|---|---|---|
| Managing Partner / Admin | Owner | Govern access, view all deals, final approval authority |
| Senior Analyst | Senior Analyst | Review, edit, resolve disputes, approve reports |
| Junior Analyst | Analyst | Upload documents, run analysis, annotate findings |
| Client-side Observer | Viewer | Read-only access to approved reports |

### Technology Choices Summary

| Concern | Choice | Why |
|---|---|---|
| Backend | FastAPI (Python 3.12) | Async-native, typed, LangGraph-compatible |
| Database | PostgreSQL 16 + pgvector | One system for relational data AND vector search |
| Files | MinIO (S3-compatible) | Files never in DB; swap to AWS S3 with one config change |
| Queue | Redis + ARQ | Long jobs survive restarts; workers scale independently |
| Cache | Redis | Embedding cost reduction; research deduplication |
| Agents | LangGraph | Stateful ReAct loops; full graph control |
| ML | XGBoost + SHAP | Tabular financial data; explainable predictions |
| Observability | LangSmith + Sentry + structlog | Full AI trace + error tracking + structured logs |
| Frontend | React 18 + Vite + Tailwind | Fast SPA; no unnecessary complexity |
| Auth | JWT + bcrypt + vault | Standard, stateless, production-grade |

## Constitution Prompt (run once with `/speckit.constitution`)

```
DealRoom AI is a production-grade multi-tenant SaaS platform for AI-powered
due diligence. Apply these non-negotiable principles throughout every phase:

SECURITY
- Every table has tenant_id. All queries filter by tenant_id — no exceptions.
- Deal room access requires explicit membership in deal_room_members.
- Document access requires a grant in document_permissions.
- Every state-changing endpoint calls log_event() before returning.
- audit_log is INSERT-only. The app DB user has no DELETE or UPDATE on it.
- Long tasks (ingestion, analysis) go through ARQ — never FastAPI BackgroundTasks.
- No secrets in source code. All credentials loaded from vault/.env at startup.
- No file content in PostgreSQL. Files go to MinIO. Only the object key is stored.
- Watermark every document download with user name + timestamp before streaming.

CODE QUALITY
- Python: type hints everywhere. Pydantic for all I/O. No bare dicts in function signatures.
- Every public function has a docstring explaining what it does and what it returns.
- All DB access through repository classes — no raw queries in routers.
- All routers are thin: validate input, call a service or repository, log event, return.
- Tests required for: auth, tenant isolation, agent outputs, ML endpoint, approval gate.
- CI must pass (tests + ML eval gate + prompt regression) before any merge to main.

ARCHITECTURE
- API versioning: all routes under /api/v1/.
- Health endpoints: GET /health (liveness) and GET /ready (readiness with DB+Redis check).
- Pagination on all list endpoints: ?page=1&page_size=20, response includes total count.
- Structured JSON logging via structlog in production. Human-readable in development.
- Sentry for error tracking. Every unhandled exception is captured with request context.
- Rate limiting on all public endpoints via slowapi.
- Environment validated at startup — app refuses to start if required vars are missing.

PRODUCT
- Empty states on every list view — never a blank screen.
- Loading indicators for all async operations — never a frozen UI.
- Error messages are human-readable and actionable — never raw stack traces.
- Approved reports are permanently read-only — enforced at the repository layer.
- The approval gate is enforced at the API layer — it cannot be bypassed from the UI.
- Export is only available on approved reports.
```

## Phase 2 — Document Ingestion Pipeline

### Spec Kit Prompt
```
/speckit.specify
Build the document upload and ingestion pipeline for DealRoom AI. Files are
uploaded to MinIO (never to PostgreSQL). An ARQ background worker processes
each document: parse with pdfplumber (tables as atomic structured chunks,
prose via recursive text splitter 512 tokens / 64 overlap), classify document
type via a lightweight LLM call, generate embeddings using text-embedding-3-small
with a Redis cache (SHA-256 hash key, 7-day TTL, batch of 100), store chunks in
pgvector with HNSW index. Default document_permissions granted to all deal room
members on upload. Job status pollable via GET /api/v1/jobs/{job_id}/status.
Watermarked download endpoint enforces can_download permission and logs
document.downloaded to audit trail.
```

### Key Models
```
documents (minio_key, arq_job_id, status), document_chunks (pgvector, HNSW index)
```

### Acceptance Criteria
- [ ] Upload a real PDF → file appears in MinIO console with correct tenant/deal path
- [ ] ARQ worker picks up job → `docker compose logs worker` shows processing
- [ ] `document_chunks` table has rows with non-null embeddings
- [ ] Poll `/api/v1/jobs/{id}/status` — transitions: queued → in_progress → complete
- [ ] Redis has `emb:text-embedding-3-small:{hash}` keys
- [ ] Upload same document twice — second ingestion hits cache (fewer OpenAI calls in logs)
- [ ] Download endpoint returns watermarked PDF with user name + timestamp on every page
- [ ] Analyst with `can_download=false` gets 403 on download

## Phase 5 — RAG & Synthesis Agent

### Spec Kit Prompt
```
/speckit.specify
Build the Synthesis Agent for DealRoom AI. Hybrid retrieval: pgvector cosine
similarity (top-15) + rank_bm25 keyword search (top-15), fused via Reciprocal
Rank Fusion (k=60), re-ranked to top-5 per section. CRITICAL: the pgvector query
must join through document_permissions before the ANN operator — agents must never
surface content a user is not permitted to see. Six report sections each get their
own targeted retrieval query. Single LLM call with full context produces a
Pydantic-validated SynthesisOutput. Every factual claim in the output is linked
to a source chunk (filename + page). Post-processing checks citation coverage

report_service.py and executed via ARQ task_run_analysis.
```

- [ ] Restrict a document to owner only → re-run analysis → restricted content does NOT appear in report
